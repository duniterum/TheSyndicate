# Sprint 13 — Chronicle Entry Publication Mechanism

**Scope:** the minimal, controlled bridge `CHRONICLE ADMISSION CANDIDATE → CHRONICLE ENTRY`, with human / governance-controlled publication semantics.
**Status:** complete. `tsc` clean · 368 targeted tests pass · ownership/forbidden-language guard OK (359 files) · `/labs/chronicle-entries-preview` renders.
**This sprint is not Story, not Recognition, not Governance, not the Member Register, not a DAO.** It builds one edge and nothing more.

---

## 1. Pre-implementation challenge answer

**A) Is Chronicle Entry the next dependency-valid layer after Chronicle Admission Candidate?**
Yes. The pipeline already terminated at `deriveChronicleAdmissionCandidates()` (admission verdict: `admitted | review | held | rejected`). Admission decides *eligibility*; it does not produce a publishable institutional record. The Chronicle Entry is the immediate, dependency-valid next edge: it reads admission candidates and projects the admitted ones into draft entries. No layer was skipped.

**B) Is there a missing layer between Admission Candidate and Chronicle Entry?**
No new *layer* — but two concepts had to be modelled *inside* the entry rather than as separate upstream layers:
- **Chronicle Publication Decision** — modelled as the `ChroniclePublicationDecision` seam on each entry (baseline = pending, null reviewer/governance/timestamp). It is the human/governance act's recording slot, not an autonomous derivation layer.
- **Chronicle Entry Draft** — modelled as the `draft` publication status, the default landing state, *not* a separate object.
- **Chronicle Seal / Chronicle Version / Chronicle Governance Approval** are deliberately **not** built. Sealing and governance approval are the future human act the `publicationDecision` seam receives; versioning is represented minimally (`version` + append-only `supersedeEntry`) without a full version-history store.

**C) Is any existing code already implementing Chronicle Entry concepts?**
Partially, and we did **not** duplicate it. `src/lib/chronicle-entries.ts` holds the locked, hand-curated public `CHRONICLE_ENTRIES` (length 3) plus a six-part `validateChronicleEntry` selection gate. That is the *public, manually curated* Chronicle. The new `InstitutionalChronicleEntry` is a **separate derived type** that maps onto the locked shape only at the future human-publish step — so the public Chronicle keeps exactly one validator and the locked entries are never touched.

**D) Architectural risk that would make Chronicle Entry premature?**
One real risk: the genesis-seed institutional facts (chapter opened, first/second artifact) **factually overlap** the three locked, hand-curated Chronicle entries. Auto-publishing derived entries would double-record the same fact in public. The mitigation *is the design*: derivation is **draft-only**, publication is a human/governance act, and nothing is written to public `/chronicle`. With that constraint the layer is not premature.

## 3. ChronicleEntry object shape

`InstitutionalChronicleEntry` (`src/lib/chronicle-entry-registry.ts`):

```ts
type InstitutionalChronicleEntry = {
  id: string;                                  // chronicle-entry:<sourceAdmissionCandidateId>
  // Lineage (carried verbatim, never re-derived)
  sourceChronicleAdmissionCandidateId: string;
  sourceInstitutionalRegisterEntryId: string;
  sourcePromotionDecisionId: string;
  sourceTxHash?: string;                       // on-chain anchor, inspection only, no magnitude
  // Classification (Chronicle-domain — the candidate's PROPOSED Chronicle register/category)
  register: "protocol-institutional";
  category: ChronicleCategory;
  // Copy (verbatim, re-validated — never regenerated here)
  title: string;
  summary: string;
  // Chronology anchor
  chronology: { date: string | null; block: number | null; txHash: string | null };
  // Verification + publication
  verificationStatus: "verified" | "coverage-limited" | ...;   // carried from candidate
  publicationStatus: "draft" | "published" | "held" | "rejected" | "superseded";
  publicationDecision: {
    verdict: "pending-human-review" | "human-approved" | "governance-approved" | "rejected";
    reviewer: string | null;        // null until a real human is recorded
    governanceRef: string | null;   // null until a real governance reference exists
    decidedAt: number | null;       // null for a deterministic baseline
    note: string;
  };
  createdFrom: string;              // the promotion rule bucket the source fact derived from
  // Audit + durability
  copyViolations: string[];
  lineage: readonly string[];      // full upstream trail, with this entry's id prepended
  version: number;                 // starts at 1; supersedeEntry increments
  supersedes: string | null;       // null for an original entry
  derivedAt: number | null;        // null — deterministic baseline, recomputed each render
};
```

**Deviations from the spec field list, by design:**
- **`sourceBlock` omitted.** It is not carried through the admission candidate, so a dedicated field would be permanently `undefined` — dishonest. The on-chain anchor is represented honestly via `chronology.block` (currently `null`) and `sourceTxHash`/`chronology.txHash`. Threading the real block height is captured as a maintainer follow-up (§10, §14).
- **`sourceChronicleReviewCandidateId` / `sourceMemoryCandidateId` / `sourceSignalId` / `sourceEventId`** are the spec's *"if carried"* fields. The admission candidate carries the full trail as a **`lineage` array**, not as separate ids, so the complete provenance lives in `lineage` rather than as four mostly-redundant scalar fields.
- **`register`/`category`** are the candidate's **proposed Chronicle** classification (`proposedChronicleRegister`/`proposedChronicleCategory`), not the institutional register's category — the admission layer already separated those two vocabularies.

## 4. Publication statuses

Strict union: **`draft | published | held | rejected | superseded`**.

`resolvePublication(admissionStatus, copyViolations)` — **copy > rules** precedence:

| Input | Result | Reason |
| --- | --- | --- |
| any carried copy violation | `rejected` | re-validated copy failed; recorded, not published (copy outranks the admission verdict) |
| `admitted` + clean copy | `draft` | eligible; awaiting a human / governance publish act |
| `review` | `held` | not a final entry |
| `held` | `held` | retained, never dropped |
| `rejected` | `rejected` | recorded, not published |

**`published` is unreachable from the deriver.** `resolvePublication` can only ever emit `draft | held | rejected`. `published` exists in the union solely as the target state of a *future human/governance act*; `superseded` is reachable only through `supersedeEntry`. Tested across all four dispositions for both the function and the deriver, and across the full genesis seed (0 published).

---

## 5. Human / governance control model

Publication is a human / governance act the derivation layer **never performs**. The seam:

- Every derived entry carries `baselinePublicationDecision()` → `verdict: "pending-human-review"`, `reviewer: null`, `governanceRef: null`, `decidedAt: null`. **No reviewer is invented, no DAO approval is implied, no governance is faked.**
- The decision is modelled as *data* (a typed seam), not a stub function, so a future human-approved / governance-approved / manually-sealed flow has an explicit, honest place to record itself — and the deterministic baseline keeps the derivation pure (recomputed each render, no fabricated timestamps).
- Auto-publish is structurally impossible from this layer: the only path to `published` is overwriting the decision externally; the baseline derivation cannot reach it.

## 8. Versioning / amendment rules

- Chronicle entries are **append-only**. Originals are immutable.
- A mistake is corrected by a **superseding** entry, never by silent in-place editing: `supersedeEntry(original, patch)` returns `{ superseded, replacement }` where `superseded` is a frozen copy of the original marked `superseded`, and `replacement` carries `version + 1` and `supersedes = original.id`. The original object is returned untouched (tested).
- Implementation is intentionally **minimal/metadata-only** (`version` + `supersedes` + the pure helper) — no full version-history store, which would be premature before a human-publish flow exists.

## 10. Maintainer intelligence

1. **Existing structure to reuse:** the locked `ChronicleEntry` type and its six-part `validateChronicleEntry` gate in `chronicle-entries.ts` — map onto it at the human-publish step instead of duplicating the gate.
2. **Risk in converting Admission → Entry:** the genesis-seed facts factually overlap the three locked public entries; publishing without a dedup/curation pass would double-record. Hence draft-only + human publish.
3. **Missing field future Story would need:** a real chronology time axis. `chronology.date`/`block` are `null` today because the pipeline carries only transaction anchors, not block timestamps; Story needs an ordered date/sequence to narrate over.
4. **Low-scope improvement that fits this sprint:** thread the source block height from the register entry through the admission candidate so `chronology.block` can be populated (a one-line addition at the admission layer; deferred here to avoid changing the upstream candidate shape).
5. **Hardening note for the future human-publish/correction flow:** `supersedeEntry` spreads the original, so a replacement inherits the original status + decision and does not re-validate patched copy. Harmless today (no production caller; supersession only ever begins from a draft baseline), but the eventual publish flow must reset the decision to baseline and re-validate patched copy before any `published` state.

## 14. Remaining risks

1. **Genesis overlap (managed, not eliminated):** derived genesis facts overlap the three locked public entries. Safe today because nothing publishes; a dedup/curation pass is required *before* any human-publish flow writes to public `/chronicle`.
2. **`supersedeEntry` re-validation gap (latent, medium):** the correction helper inherits the original status/decision and does not re-validate patched copy. No production caller today; the future publish flow must reset the decision to baseline and re-validate patched copy before `published`. Recorded as maintainer note #5.
3. **Block-anchor incompleteness (low):** `chronology.block`/`date` are `null` until block height/timestamp is threaded through the admission candidate (maintainer note #4). Entries are tx-anchored but not yet time-ordered — a prerequisite for Story.
4. **Activity back-references deferred (low):** no institutional-memory metadata was added to live Activity rows; recommended as a separate later pass rather than risking overload of the raw stream.

None of these block the sprint objective: the controlled `Chronicle Admission Candidate → Chronicle Entry` bridge exists, is draft-only, copy-gated, adjacency-clean, append-only, and publishes nothing without a human / governance act.
