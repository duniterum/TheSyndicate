# NFT Archive — Metadata Philosophy

**Status:** FROZEN PHILOSOPHY · B1 LOCKED · ID 1 LIVE ONCHAIN SVG
**Doctrine:** SYN is the seat. NFT Artifacts are the memory.

This document freezes the metadata philosophy for the NFT Archive. The
deployed Archive1155 contract now uses this model for ID 1; future artifact
families must continue to satisfy it.

## 1. Metadata priorities (in order)

1. **Permanence** — an artifact's meaning must survive the website,
   the founder, and the developer.
2. **Verifiability** — every field must be checkable against on-chain
   data or against a frozen on-chain renderer.
3. **Autonomy** — no dependency on a live website, indexer, or
   centralized image host for the artifact's core meaning.
4. **Readability** — humans should understand the card without docs.
5. **Marketplace compatibility** — must render correctly on Joepegs,
   OpenSea, Campfire, etc., without bespoke integration.

---

## 2. V1 direction (LOCKED, decision B1)

**V1 default: fully on-chain JSON metadata and on-chain SVG image.**

- The contract stores per-id Artifact Definitions (templates + dynamic
  data) and composes both the JSON and the SVG at call time.
- `uri(id)` returns a `data:application/json;base64,…` payload.
- The JSON's `image` field is a `data:image/svg+xml;base64,…` payload
  generated by the contract.
- **No external hosting required.** No IPFS pin to maintain. No Arweave
  upload to fund. No website fallback that can rot.

This is consistent with the doctrine ("memory") and with the founder's
stated goal of minimum external dependency.

See `docs/design/nft-archive-onchain-engine-reference.png` for the
intended JSON + SVG shape.

---

## 3. Renderer modes (binding)

The contract supports exactly three renderer modes per token id:

| Mode             | V1 use                                              | Behavior of `uri(id)`                                                                 |
| ---------------- | --------------------------------------------------- | ------------------------------------------------------------------------------------- |
| `ONCHAIN_SVG`    | **Default for all activatable V1 token ids.**       | Returns `data:application/json;base64,…` with on-chain-generated SVG `image` data URI. |
| `EXTERNAL_URI`   | **Reserved** for future complex/rich art only.      | Returns a stored `ar://` or `ipfs://` JSON URI. **Not used by any V1 token id.**       |
| `NONE` / `LOCKED`| Configured but not yet ready to render.             | Reverts (`URINotReady`) — token id is visible as configured, not yet renderable.       |

V1 mapping (see `NFT_ARCHIVE_TOKEN_CATALOG_V1.md` for full detail):

- ID 1 The First Signal → `ONCHAIN_SVG`
- ID 2 Seat Record → `LOCKED` (pending eligibility, future ERC-721 likely)
- ID 3 Patron Seal → `ONCHAIN_SVG`
- ID 4 Heart Signal → `ONCHAIN_SVG` or `LOCKED` until discovery flow is active
- ID 5 Genesis Sealed → `ONCHAIN_SVG` or `LOCKED` until genesis closes
- ID 6 First Liquidity Event → `ONCHAIN_SVG` or `LOCKED` until LP verified
- ID 7 First Routing Signal → `ONCHAIN_SVG` or `LOCKED` until routing verified
- ID 8 Legacy Era I → `LOCKED`

`EXTERNAL_URI` is enabled in the contract but **no V1 token id ships in
that mode.** Switching a token id to `EXTERNAL_URI` requires explicit
founder approval and is blocked once that id's Artifact Definition is
frozen.

---

## 4. Why not blindly use IPFS / Arweave for V1

- **External pin rot risk.** IPFS pins lapse if not funded; Arweave
  requires a one-time payment correctly anchored.
- **Operational burden.** Each new artifact would require an off-chain
  upload pipeline, a metadata signing step, and a permanence audit.
- **Trust regression.** Off-chain images make the artifact's meaning
  depend on a service the founder does not control.

On-chain SVG removes all three concerns at the cost of bytecode/gas at
deploy time and a slightly heavier `uri(id)` call. The V1 artifacts
(certificates, seals, simple cards, pixel marks) are exactly the kind
of art that fits within the on-chain renderer's budget.

---

## 5. Why not put raw image bytes on-chain

- Raw PNG/JPEG bytes are prohibitively expensive even on Avalanche.
- They cannot be parameterized per token (serial, wallet, chapter).
- A template + dynamic fields renderer is cheaper, more flexible, and
  produces visually consistent cards across all serials of a token id.

The right path is **template + data + on-chain SVG composition**, not
"upload images on-chain."

---

## 6. Invariants — what must remain true

- **No fake metadata.** Every field traces to a real on-chain value or
  a frozen template constant.
- **No mutable meaning after activation.** Once a token id is activated
  and `freezeArtifactDefinition(id)` is called, neither the renderer
  logic for that id nor the dynamic field semantics may change.
- **No website-only metadata.** The website MAY mirror metadata for UX,
  but is never the source of truth.
- **No promotional traits.** No `trait_type` implying yield, dividend,
  revenue share, governance rights, Vault entitlement, or guaranteed
  appreciation.
- **No identifying PII.** Wallet address is the only identity field.
- **Marketplace-safe.** Required fields: `name`, `description`, `image`,
  `attributes[]`, optional `external_url`. No non-standard top-level
  keys that break Joepegs/OpenSea rendering.
- **No `https://` in V1 default path.** A `ONCHAIN_SVG` token id MUST
  NOT resolve through an `https://` URL.

## 7. Canonical metadata shape (V1, on-chain rendered)

The contract composes this on-chain at `uri(id)` time. The example
below shows the decoded payload — actual `uri(id)` returns the
base64-encoded form prefixed `data:application/json;base64,`.

```json
{
  "name": "The First Signal #000142",
  "description": "Chapter I Artifact. Collectible record of an early Syndicate chapter. Confers no financial rights.",
  "image": "data:image/svg+xml;base64,PD94b…",
  "external_url": "https://thesyndicate.money/archive",
  "attributes": [
    { "trait_type": "Category",         "value": "Chapter Artifact" },
    { "trait_type": "Token ID",         "value": 1 },
    { "trait_type": "Chain",            "value": "Avalanche C-Chain" },
    { "trait_type": "Renderer",         "value": "ONCHAIN_SVG" },
    { "trait_type": "Status",           "value": "Active" },
    { "trait_type": "Rights",           "value": "Collectible record only" },
    { "trait_type": "Financial rights", "value": "None" },
    { "trait_type": "Visual family",    "value": "Artifact Card" },
    { "trait_type": "Chapter",          "value": "I — The Beginning" },
    { "trait_type": "Serial",           "value": "000142" }
  ]
}
```

Notes:
- `image` is a data URI. No external host.
- `external_url` is a convenience link; the artifact remains meaningful
  even if the site goes offline.
- `attributes` are descriptive only. No financial or governance traits.
- The `Renderer` attribute exposes the active `RendererMode` so
  marketplaces and indexers can verify the artifact's permanence model.

## 8. SVG content contract (binding)

Every on-chain SVG rendered by V1 MUST include, as readable text:

- `The Syndicate` wordmark/seal
- Artifact name
- Category badge
- Token ID
- Serial number (if applicable; for capped supply artifacts)
- `Avalanche C-Chain`
- `Collectible record only`
- `No financial rights`

The exact glyph and palette derive from the per-id Artifact Definition
(`visualFamily`, `primaryColor`, `accentColor`, `iconId` / `templateId`).
See `NFT_ARCHIVE_VISUAL_SYSTEM_V1.md` for visual families.

---

## 9. Banned wording in metadata (constitutional)

Never appears in `name`, `description`, or any `trait_type` / `value`:

`investment`, `yield`, `dividend`, `revenue share`, `profit`,
`guaranteed value`, `Vault entitlement`, `holder rights`, `reward`,
`utility` (financial sense), `airdrop` (as promise), `passive income`,
`appreciation` (as promise), `ROI`.

## Final Seat Record Architecture Decision (✅ LOCKED 2026-06-06)

Seat Records are **NOT** active ERC-1155 artifacts in `SyndicateArchive1155` V1.
They will ship later in a **separate ERC-721 contract**
(working name `SyndicateSeatRecord721` / `SyndicateSeatRegistry721`)
once membership eligibility is enforceable on-chain.

Token ID 2 in `SyndicateArchive1155` is a **reserved + disabled pointer**:
`active = false`, `ownerOnly = true`, `rendererMode = NONE`,
`maxSupply = 0` (LOCKED / NOT MINTABLE), `walletLimit = 1`,
`priceUsdc = 0`. All mint paths revert; `uri(2)` reverts.

Contract-wide V1 rule: **`maxSupply == 0` means LOCKED / NOT MINTABLE**.
It does **NOT** mean unlimited. No V1 artifact is unlimited.

Allowed wording: *"Seat Records are reserved for a future ERC-721
identity contract."* / *"Archive1155 V1 records collectible protocol
artifacts."* / *"ID 2 is reserved and disabled in V1."* /
*"SYN is the seat. Artifacts are the memory."*
