# Engine completeness ≠ pipeline completeness

The canon's frozen 5-layer model (TRUTH→EVENTS→SIGNALS→MEMORY→STORY + Adjacency Law, `docs/canon/05`) is a **one-directional refinement PIPELINE, not a closed self-reinforcing engine** — it has no feedback edge back to acquisition. An engine that generates trust/participation/retention/growth/value needs a closing loop: STORY/DIFFUSION → new IDENTITY → EVENTS. Today each member is an isolated transaction, not a node that produces the next member.

**Why:** "Money is proof the machine is alive" (05 §4.5) is true but ≠ self-reinforcing. Inflow alone doesn't compound members. Diffusion is treated as a *feature*, not a *layer*, which is why it's under-built as a system.

## How to apply
When asked "is the protocol done / what's next," **audit the LOOP, not the page list.** Top-3 leverage order:
1. **Close the DIFFUSION loop** — cheapest; infra already LIVE, just not wired into one loop (post-purchase identity moment → share card → verifiable landing).
2. **Minimal SIGNALS engine** — the keystone the constitution waits on; hand-curation (static MOMENTS array + locked chronicle-entries) breaks at a few hundred events, so Signals is **mandatory at 1,000+ members**.
3. **Non-money RECOGNITION** — compounds retention; honors the money≠prestige guardrail (a MEMORY output of Signals).

Governance/Referral-contract are intentionally downstream/reserved — do not put them in a "next" list.
