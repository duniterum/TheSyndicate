Founder constitutional ruling (recorded in canon 05 §2.1; closes the FS-1 open question in canon 07):

**SIGNALS read EVENTS only. There is NO Adjacency carve-out for SIGNALS reading MEMORY — now or later.**
Chronicle appearances and Recognition milestones are MEMORY *outputs*, not Signal *inputs*.

- Correct flow:   EVENT → SIGNAL → RECOGNITION / CHRONICLE
- Forbidden flow: MEMORY → SIGNAL  (circular; weakens the anti-drift model)

**How to apply.** If a future Signal needs a fact that currently lives only in MEMORY (e.g. Chronicle
appearance, recognition milestone, or the "Historic" signal type in 05 §3.2), DO NOT read MEMORY.
Re-express the underlying source as an EVENT first, then derive the Signal from that EVENT. The frozen
chain stays one-directional: TRUTH → EVENTS → SIGNALS → MEMORY → STORY. The only sanctioned cross-layer
carve-out is the pre-existing one (MEMORY may ingest EVENTS directly for raw Activity).

**Why:** allowing SIGNALS → MEMORY creates a circular architecture (SIGNALS → MEMORY → SIGNALS) and breaks
the whole layered anti-drift design. This was a deliberate founder ruling, not an implementation detail —
do not "optimize" it away by letting a future Signal Engine peek at Chronicle / Recognition.
