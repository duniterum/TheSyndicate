---
name: Hero radial throne/ring layout
description: Why the ProtocolHero radial ring stacks seat-text ABOVE the ring, throne alone on the top arc, and a small central-number clamp.
---

The hero radial ring (`ProtocolHero` `RadialStage`, ~400px wide at the 3-column breakpoint) is too compressed to stack a throne + "You are visitor / Seat available" text + the large central "$X / LIVE CAPITAL FLOW" value vertically. They collide.

**Composition / responsive structure (3-row CSS grid).** `ProtocolHero`'s grid renders children in DOM order = the founder's MOBILE order: `[HeroLeft, HeroEngine, EntryRail, HeroRight, ChapterStrip]` (headline+CTAs → throne+$engine → entry preview → overview+latest-activity → current chapter). Desktop is reconstructed via **explicit `xl:` grid placement** that overrides DOM order: row1 = HeroLeft(col1)/HeroEngine(col2)/HeroRight(col3); row2 = ChapterStrip (col1 span2, under left+center); row3 = EntryRail (full-width, col1 span3). **The 3-col layout keys on `xl` (1280), NOT `lg`** — below 1280 the grid is `grid-cols-1` and everything stacks in DOM/mobile order (spec: tablet stacks, desktop ≥1280 is 3-col).
- Right column is **Protocol Overview → Latest Activity ONLY**; Current Chapter is a wide HORIZONTAL strip (`ChapterStrip`), never a card in the narrow right column.
- The right column is narrow, so the Protocol Overview's internal 3-col (`xl:grid-cols-3`) only fits at 1280 after: column ratio `1.1fr/1.78fr/1.12fr` (right bumped from 0.95 once the chapter card left it), Stat values `text-lg` (not `text-xl`), `pl-3`, `gap-x-4`. Reverting to text-xl/0.95fr makes adjacent overview values overlap ("$10M$10M") at 1280.
- Each hero child accepts an optional `className` merged onto its root, used only to carry the `xl:` placement strings.
**Why:** founder "HERO COMPOSITION FIX" — engine stays centered+dominant; chapter progression bridges hero→entry rail in the empty horizontal band; no card may crowd the ring.

The reference mockup (image 3) resolves it, and the code now mirrors it:
- Seat text ("You are visitor / Seat available") lives **above** the `aspect-square` ring (`hidden sm:block`), not inside its top — this frees the ring's top zone.
- The throne (`/hero/throne.png`, ~square 560×546) sits **alone** on the top arc (`top-[1.5%]`), sized as a **percentage of the ring** (wrapper `w-[24%]`, img `w-full`) — NOT fixed px. **Why:** the ring width ranges hugely across breakpoints (~420px @1280 → ~640px capped at wide), so any fixed-px throne over-dominates the narrow ring (a 138px throne ≈ 33% of the ring @1280, overlapping the center value) yet looks small at 1920. A `%` keeps it proportional everywhere; ~24% ≈ the founder's "identity anchor" size (+~26% over the old ~19%-at-wide). Earlier non-monotonic fixed widths (80/72/84/92…) are obsolete.
- The central value clamp is `clamp(3rem,0.7rem+3.9vw,5.5rem)` — a **steep-vw / low-base** shape so it stays ~3.2rem at the tight lg=1024 (collision-safe) but ramps to ~5.5rem at wide (~+30%, the founder's "$ should dominate" ask). The value block also carries `pt-[6%]` so its centered stack drops ~3% to clear the bigger throne. The original `clamp(4.9rem,…,7.25rem)` was ~27% of the ring and forced a collision; do not return to a high-base clamp.

**Why:** at 1280 the ring is ~400px; a big number's centered value-stack tops out near ~25% of the ring, leaving no room for a throne + seat text at the top.

**How to apply:** if re-enlarging the hero number or moving the seat text back inside the ring, expect throne/number collision — re-test at the **xl breakpoint (exactly 1280, the tightest 3-col; below xl the hero is single-column)**, not just wide screens.

Related: `AvalancheMark` is the exported official-logo SVG in `HeaderWalletChip` (red circle + evenodd white AVAX cutout), reused by `ProtocolHero` `ProofMini`; old hand-drawn `AvalancheIcon`/`ThroneIcon` removed.

**Mountain bg = `public/hero/cervin.jpg` (real Matterhorn/Cervin photo), shown in BOTH themes.** This OVERRIDES the earlier dark-only doctrine: the old `public/hero/mountain-bg.jpg` (generated, `hidden dark:block`) was replaced at founder request with a real Swiss peak, and `HeroAtmosphere` now renders it in light AND dark (`opacity-[0.5] dark:opacity-[0.6]`, NOT `hidden dark:block`). Scrims use `var(--background)` color-mix so the photo washes to ivory in light / obsidian in dark — light mode shows a subtle warm mountain field (rocky foreground low-center), not the old no-photo ivory. Do NOT re-hide the photo in light mode.

**Homepage header→hero blend + alignment doctrine.** The global `Header` (rendered by `PageShell`, NOT gated by `hideHeader` — that flag only hides the page-title block) is the alignment reference for the hero. Blend rules: `HeroLeft` carries no status pill so the headline sits high; the homepage suppresses the strips between header and hero via `PageShell` props `hideDemoBanner` + `hideIdentityRibbon` (default false; homepage only); the grid is `items-start` so the headline column and right metrics panel top-align under the header; the `HeroEngine` root is `lg:self-start` (NOT self-center) so the throne engine TOP-aligns to the column top. **Why:** founder ruled the throne must sit at ~the same height as the headline top and the Protocol Overview card top — `self-center` floated it to the middle of the very tall grid ("mid-page"), which was the #1 complaint. Seat text renders above the ring (mb-4), so the throne crown lands ~60px below the top line — this matches the founder's sketch (Seat text → throne → $), intended, not a defect.
**Why the header has a `wide` prop:** header + hero MUST share one container width or the hero overhangs the header on wide screens (founder viewport ≈1920). But the ring lives in the CENTER grid column, so a "bigger ring" is impossible inside the global 1280 (`max-w-7xl`) header — the column caps the ring at ~560. Resolution: `Header` takes a homepage-only `wide` prop (`PageShell.headerWide`) widening BOTH header and hero to `max-w-[1480px]`; every other page stays `max-w-7xl`. 1480 is chosen so the center column finally reaches the `max-w-[640px]` ring cap (cap is meaningful, not dead), yielding a ring bigger than the prior ~580 while header≈hero edges stay aligned; cost = ~100px header overhang past the 1280 lower homepage sections on very wide screens (accepted).
**Durable geometry gotchas:** the left column fr must stay ≥~1.1 or the "economy." headline overflows at the xl (1280) 3-col breakpoint; the right panel (3 stacked cards) is taller than the viewport — with the engine now `self-start` the ring sits at the top, so its bottom nodes still extend below a 720px capture fold (expected: the ring is ~640px tall; a capture-tool artifact, NOT a bug, do NOT shrink the ring to "fix" it). No test pins the pill / DemoBanner / IdentityRibbon / hero layout classes.

**Darkening the hero green:** to make the ring green "darker," use a hero-local `GREEN_DEEP = color-mix(in oklab, var(--success) ~78%, #052e1a)` (mix toward forest green so it ADAPTS in both themes), applied to the ring stroke / `heroHub` gradient / soft halo — **never** touch the global `--success` token (it drives LIVE pills site-wide). Keep the focal central value ("$X" / LIVE CAPITAL FLOW / USDC ROUTED) at full `var(--success)` brightness so it still pops; a darker surround actually improves its figure-ground contrast. Nested color-mix in inline style / SVG `stopColor` is fine (the codebase already hard-depends on color-mix). **Why:** founder wanted a deeper green circle without dimming the headline number or recoloring every status pill.

**Green inner ring radius ceiling:** in the 400×400 `RadialStage` viewBox the green hub+stroke circle can grow to ~r=114 (28.5% radius) — the gold rings stay at r=132/166/190, and every lane/CTA node (VAULT/LIQUIDITY/OPERATIONS/CHRONICLE/JOIN/MEMBERSHIP) sits at ≥34% radial distance, so a size-13 icon's inner edge (~30%) clears r=114 by ~2%. Do NOT exceed ~r=118 or nodes crowd. Throne at top-[1.5%] sized at ~24% of the ring (bottom ≈25% of ring height) clears both the ring top edge and the value-block label below it.

**FINAL composition+perf pass (supersedes some numbers above — trust the code, not these notes, for exact clamp/%):** the founder's "engine too low" was cured by raising the focal stack (`justify-start` + `pt-[~30%]`), enlarging the routed-value clamp, growing the throne (~26% at top-2%) with a radial gold glow div behind it (so it reads integrated, not pasted), and spreading all **6 orbit nodes onto a symmetric outer ring** (left/right pairs at ~31/56/84% heights) instead of clustering them.

**abspos-node width trap (non-obvious CSS).** An `EngineNode` positioned with only `left-[N%]` (no width, no `right`) has its available width constrained to `container − N%`. At `left-[90%]` that leaves an ~8% sliver, so a 2-word value ("With USDC") wraps to two lines and collides with the node below. **Fix = `whitespace-nowrap` on the node wrapper** (the `-translate-x-1/2` still centers it), NOT widening the column. **How to apply:** any near-edge (≤12% or ≥88%) absolutely-positioned label needs `whitespace-nowrap`.

**Reduced motion must be gated in THREE independent places** — missing any one leaves motion running for `prefers-reduced-motion` users: (1) the orbit/lane traveling dots via `chainLive = blockNo!==undefined && !reducedMotion`; (2) `AnimatedNumber`'s count-up (snap straight to target when reduced — lives in `Primitives.tsx`, used app-wide); (3) the **decorative SVG dashed-ring `<animateTransform rotate>`**, which is unconditional unless wrapped — gate by calling `usePrefersReducedMotion()` *inside* `RadialStage` (don't reuse `chainLive`, which would also stop the ring whenever `blockNo` is merely loading). **Why:** the SVG ring rotation is the one easy-to-miss animation because it's not React-driven.

**Hero perf doctrine:** ship the hero bitmaps as **WebP** (`cervin.jpg→cervin.webp`, `throne.png→throne.webp`; keep originals), `<link rel="preload" as="image">` the LCP background in the route head, and give the throne `<img>` explicit `width/height` (natural 560×546) so it reserves aspect-ratio and avoids CLS. The big green `—` placeholder when `routedValue===undefined` is EXPECTED in dev (Avalanche RPC is CORS-blocked on localhost/preview); live data renders in prod — do not treat it as a layout bug.

**Connected-header fit @1280 is already solved:** below `2xl` the connected `HeaderWalletChip` shows a compact `…XXXX` (last 4) with the caret hidden, and disconnected shows just "Connect" — both states fit the 1280 width budget by design. Don't "fix" header overflow by shrinking the hero.

**Burn monument (seat-top / burn-bottom).** A `BurnMonument` (animated CSS flame + "Burned supply / {fmtCount(burnedSyn)} SYN / Proof of Fire") is the LAST child of `HeroEngine`, so it renders directly beneath the throne/ring in the center column — the founder's "seat on top, burn at the bottom" (beginning↔end of supply). It is mountain-integrated: NO card/border/panel/CTA/link. Value = live `useSynSupply().burned` (= `balanceOf` dead address); degrades to `—` + status (PENDING) when undefined, and shows "Proof of Fire" ONLY when known.
- **Doctrine term is "Proof of Fire", never "Proof of Burn"** — even though the reference mockup image literally says "Proof of Burn", spec defers to canon vocabulary (105 "Proof of Fire" refs in docs/canon+src, "Proof of Burn"=0). FLAME palette const = `#F97316` (flame orange); it's burn-only (monument + overview burn stat/pill/SVG), so harmonizing it is low-risk.
- **The flame is a layered inline SVG (NOT div blobs — that read as a "candle").** Three flame `<path>`s (outer amber / inner gold / white-hot core) with vertical `linearGradient` fills (deep-ember `#B45309` base → `#F97316` → warm-gold `#E3A92B` → hot-core `#FDE68A`; dark red `#7C2D12` only in the base stop + drop-shadow), a flex-centered radial glow div behind, and 8 embers via 3 keyframe variants (`syn-ember-l/r/c`). Burned number = gold→ember `background-clip:text` gradient.
- **Centering supersedes the old `translateX(-50%)`-in-keyframe hack.** The SVG is centered by a flex wrapper (`items-end justify-center`), and each flame `<g>` flickers via `transformBox:fill-box` + `transformOrigin:"50% 100%"` so scaleY anchors at the flame BASE — so the keyframes (`syn-flame-a/b/c`, `syn-glow-pulse`) are pure `scaleY`/`translateX`/opacity with **NO centering offset**. Do NOT reintroduce `translateX(-50%)` into these keyframes.
- **Reduced motion is gated TWICE (both required).** (1) JS: `usePrefersReducedMotion()` drops embers entirely + sets each inline `animation:undefined`; (2) CSS first-paint backstop: a `@media (prefers-reduced-motion: reduce)` rule forces `animation:none !important` on `.syn-burn-flame *` — needed because the hook returns false until mount, so without the CSS the flame animates on first paint for reduced-motion users. Keep both; the class `syn-burn-flame` is the CSS hook.
- Like the ring's bottom nodes, the burned-supply VALUE text sits just below a 720px capture fold (the center column = throne+ring+burn is taller than one fold); the value is ALSO mirrored in the right Protocol Overview ("Burned 2,000 SYN / Proof of Fire"). Expected, do NOT shrink the engine to pull it up.

**Orbit = exactly 5 labels; `NodeLabel` skips falsy values.** To hit the reference's 5-node orbit, the "Join / With USDC" `EngineNode` was removed and Membership Sale's value set to `""`. `NodeLabel` now renders the sub-value line only when the value is truthy, so `value=""` ⇒ a title-only node. **How to apply:** to show a node as title-only, pass an empty-string value (don't delete the value line); real formatted values (`—`, `$0.00`) are truthy and still render.
