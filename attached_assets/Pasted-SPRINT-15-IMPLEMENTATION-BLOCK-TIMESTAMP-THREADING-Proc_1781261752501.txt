SPRINT 15 IMPLEMENTATION — BLOCK TIMESTAMP THREADING

Proceed carefully.

Do NOT run a broad audit.

Accepted completed chain:

Event
→ Signal
→ Memory Candidate
→ Chronicle Review Candidate
→ Promotion Decision
→ Institutional Register Entry
→ Chronicle Admission Candidate
→ Chronicle Entry
→ Chronological Timeline

Current limitation from Sprint 14:

Chronology proves order by block height, but blockTimestamp is null.

Core objective:

Thread verified block timestamps into the Chronicle Chronology Layer so future Chronicle publication and future Story can use real dates without inventing them.

This sprint does NOT:

* build Story
* publish Chronicle entries
* build Recognition
* build Governance
* build AI
* create new contracts
* change chronological ordering rules
* infer dates
* estimate dates

It only adds verified timestamp metadata to already block-anchored chronology entries.

1. PRE-IMPLEMENTATION CHALLENGE

Before coding, answer briefly:

A) Is Block Timestamp Threading the next dependency-valid layer after Chronicle Chronology?

B) Is any hidden dependency missing before timestamps can be added?

C) Does the codebase already fetch block timestamps anywhere?

D) Is there any architectural risk that would make timestamp threading premature?

Keep the answer concise.

Do not perform a broad audit.

2. TIMESTAMP SOURCE

Use only verified block timestamp sources.

Preferred:

* chain RPC getBlock / block timestamp
* existing block metadata hook if already present
* existing cached block lookup if already present

Do not use:

* approximate dates
* local clock
* deployment date guesses
* explorer scraping
* hardcoded dates unless already locked as verified config

3. TIMESTAMP OBJECT / FIELD

Extend chronology metadata safely.

For block-anchored entries, populate:

* blockTimestamp
* timestampStatus
* timestampSource
* timestampConfidence
* timestampReason

Suggested statuses:

* verified
* unavailable
* pending
* error
* not-applicable

4. CACHE / PERFORMANCE

Avoid excessive RPC calls.

Rules:

* deduplicate block numbers before lookup
* cache block timestamps if there is an existing query/cache pattern
* avoid one RPC call per render if possible
* fail softly if RPC is unavailable

Do not introduce heavy indexing infrastructure.

5. FAILURE / PARTIAL STATE

If timestamp lookup fails:

* keep chronology ordering by block number
* set timestampStatus appropriately
* do not hide the entry
* do not invent a date
* do not mark date as verified

6. CHRONOLOGY RULES

Do not change ordering hierarchy.

Ordering remains:

* block number ascending
* source-entry-id tie-break

Timestamp is metadata only.

Timestamp must not become the ordering source.

7. SEEDED VS PIPELINE FACTS

Apply the same timestamp logic to:

* genesis-seeded entries
* pipeline-derived entries

No special timestamp rules for seeded facts except source integrity.

8. PUBLIC / LABS SURFACE

Update only internal/labs chronology inspection if needed.

Acceptable:

* /labs/chronicle-timeline

Show:

* block number
* timestamp status
* timestamp value if verified
* timestamp source
* reason if unavailable

Do not expose public Story.

Do not redesign public Chronicle.
