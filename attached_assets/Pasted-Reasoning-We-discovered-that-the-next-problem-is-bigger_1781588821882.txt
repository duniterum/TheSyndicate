Reasoning

We discovered that the next problem is bigger than CTA labels.

The Syndicate already has many action and guidance fragments:

* wallet gate
* protocol action registry
* guard layer
* cockpit next move
* cockpit progression
* cockpit memory
* since-you-were-away surfaces
* route final CTAs
* homepage progression teaser
* rank/milestone prompts
* archive onboarding
* member/non-member states

The risk is that we wire CTAs before understanding the user journey system.

The Syndicate is not only a sale page. It is a membership, identity, story, artifact, and progression protocol.

So before implementing CTA Architecture V1, check whether the correct next backbone is:

Identity State
→ Next Best Action
→ Protocol Actions
→ CTAs

rather than:

Protocol Actions
→ CTAs
→ later guidance

This is a focused architecture-fit task, not a broad audit.

Prompt

Pause implementation.

Use the current codebase, Full Protocol View, and existing docs.

Do not run a broad audit.

Do not implement yet unless the answer is obvious and very low-risk; first report your recommendation.

Goal:
Decide the best next architecture move for The Syndicate without forgetting existing modules or creating technical debt.

Important context:

We confirmed:

* `joinMembership` should stay as the internal action id.
* “Become a Syndicate Member” is a label/state-specific CTA, not necessarily a new id.
* Enter/connect/login is NOT membership.
* A user may connect/login and view My Syndicate before becoming a member.
* A user becomes a Syndicate Member only by buying membership through active Sale V2 and receiving SYN.
* `nextBestAction` is not itself an action; it is a selector/engine over protocol actions.
* Current guidance is fragmented across several components.
* CTA registry wiring may be premature if guidance/state logic is not unified.

Current truths to preserve:

* Sale V2 is deployed/funded/active and used by the frontend, but UNAUDITED / early.
* Sale V1 is sealed/closed/history.
* CommissionRouter is unset; referral remains pending.
* Archive1155 is artifacts only.
* SeatRecord721 is future.
* Web2 + Web3 onboarding is future, not implemented now.
* Current wallet flow remains injected/wagmi.
* Referral claims / campaign rewards / merch eligibility are future PENDING concepts only.
* No fake live actions.
* No new promises.
* No extra steps before purchase.
* No friction added.

Question to answer:

Given the actual codebase, should the next sprint be:

Option A — Canonical CTA Architecture V1
Add `enterSyndicate`, settle canonical labels/destinations, update guard conformance, then wire CTAs later.

Option B — Next Best Action Foundation V1
Create a pure selector/state model that maps user state to the best next action, then let CTA architecture consume it later.

Option C — Hybrid
Add only the minimum registry truth needed now, but start the Next Best Action foundation before wiring CTAs.

Do not guess. Check the current system and recommend the best fit.
