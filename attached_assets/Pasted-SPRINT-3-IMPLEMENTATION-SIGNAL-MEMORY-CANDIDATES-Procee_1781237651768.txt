Core objective:

Build the minimal Memory Candidate layer:

Signal → Memory Candidate

This sprint does NOT publish automatic Chronicle entries.

This sprint does NOT build Recognition.

It only creates the safe intermediate layer that decides:

“What is worth remembering?”

Hard rules:

* Memory Candidates read Signals only.
* Memory Candidates may reference source Events through the Signal.
* Memory Candidates must never read raw Events directly unless via Signal.sourceEvent.
* Memory Candidates do not invent facts.
* Every Memory Candidate must cite its Signal.
* Every Memory Candidate must preserve verification lineage:
  Signal → Event → Tx / Block / Source Fact.
* No new contracts.
* No public marketing UI.
* No ROI / yield / dividend / governance / rights language.
* No founder glorification.
* No money-as-prestige.

1. MEMORY CANDIDATE OBJECT

Create or formalize a MemoryCandidate object.

It should include:

* id
* register
* category
* subject
* sourceSignalId
* sourceEventId
* sourceTxHash if available
* sourceBlock if available
* title
* summary
* reason
* verification status
* recommendedAction
* createdFrom

2. REGISTERS

Implement two memory registers:

A) Protocol / Institutional Register

For protocol-level moments:

* genesis events
* protocol milestones
* treasury threshold crossings
* burn events
* first liquidity
* first artifact
* significant system wallet actions
* meaningful founder/system actions

B) Member / Living Register

For member-level moments:

* new member
* early member
* repeat participation
* milestone witness
* artifact participation
* continuity signals

3. CATEGORIES

Use safe categories such as:

* genesis
* membership
* treasury
* operations
* liquidity
* burn
* artifact
* milestone
* chapter
* founder-action
* system-wallet-action
* continuity

Do not create noisy categories.

4. SIGNAL → MEMORY RULES

Define deterministic rules:

* S0 routine → usually no Memory Candidate.
* S1 notable → optional candidate only if category is important.
* S2 personally significant → Member / Living candidate.
* S3 protocol significant → Protocol / Institutional candidate.
* S4 rare → Protocol / Institutional candidate.
* S5 historic → Protocol / Institutional candidate.

Rules must be deterministic and testable.

5. FOUNDER / SYSTEM WALLET RULE

Founder and system wallet actions must be handled carefully.

Every detected action may appear in Activity.

But Memory Candidate creation requires significance.

Examples:

* routine transfer → no Memory Candidate.
* first operations funding → Protocol Candidate.
* first campaign spend → Protocol Candidate.
* major Vault threshold → Protocol Candidate.
* first LP action → Protocol Candidate.
* founder burn → Protocol Candidate.
* treasury asset acquisition → future Protocol Candidate.

Do not glorify the founder.

The protocol is the subject unless the action is purely personal.
