Memory Candidate → Chronicle Candidate

“Which Memory Candidates are eligible to become permanent Chronicle entries later?”

Hard rules:

* Chronicle Candidates read Memory Candidates only.
* Chronicle Candidates do NOT read raw Events directly.
* Chronicle Candidates do NOT read Signals directly except through Memory Candidate lineage.
* Chronicle Candidates must preserve full verification lineage:
  Chronicle Candidate → Memory Candidate → Signal → Event → Tx / Block / Source Fact.
* No invented facts.
* No automatic public Chronicle publication.
* No founder glorification.
* No money-as-prestige.
* No ROI / yield / dividend / governance / rights / payout language.

2. REGISTERS

Preserve the two-register model:

A) Protocol / Institutional Register

For protocol-level history:

* genesis events
* first liquidity
* first burn
* first artifact
* first milestone
* significant treasury movement
* significant operations movement
* significant liquidity event
* future protocol asset acquisition
* future external revenue event
* meaningful founder/system action

B) Member / Living Register

For member-level history:

* first member
* milestone member
* repeat participation
* continuity
* milestone witness
* artifact participation
* future recognition-worthy member moments

3. CHRONICLE ELIGIBILITY RULES

Define deterministic eligibility.

Suggested starting rules:

* S0 → never Chronicle Candidate.
* S1 → only if category is explicitly first/new/continuity and passes coverage.
* S2 → Member / Living Candidate, review required.
* S3 → Protocol / Institutional Candidate, review required.
* S4 → high-priority Chronicle Candidate.
* S5 → historic Chronicle Candidate.

Do not auto-publish any tier.

Every candidate is review-first.

4. REVIEW STATUS

Add review workflow fields:

* needs-review
* approved
* rejected
* hold-coverage
* hold-context

Default should be:

* needs-review for normal valid candidates
* hold-coverage when scan coverage is not sufficient
* hold-context when the event is valid but meaning requires human explanation

Do not auto-approve except maybe locked genesis entries if already verified and existing.

5. FOUNDER / SYSTEM WALLET RULE

Founder and system wallet actions are important but must be disciplined.

Principle:

Founder is an actor inside the protocol, not outside it.

Founder may be:

* founder
* user
* member
* funder
* liquidity actor
* artifact holder
* future governance participant

Classify by:

* actor role
* wallet role
* action type
* protocol significance
* verification source

not by founder label alone.

Rules:

* Routine founder/system actions remain Activity / Memory only.
* Significant institutional actions become Chronicle Candidates.
* Founder/system actions must never become Chronicle only because the founder did them.
* Founder/system actions may become Chronicle when they materially affect the protocol.

Examples eligible:

* founder seeded initial liquidity
* founder performed Proof of Fire burn
* founder funded first operations campaign
* Operations wallet spent for first campaign
* Liquidity wallet executed major LP action
* Vault wallet crossed major threshold
* Treasury acquired external asset / token / company stake
* protocol received first external revenue
* major protocol-owned asset movement

Examples not eligible:

* routine wallet rebalancing
* dust transfers
* internal admin movements
* allocation shuffling
* repeated small top-ups without protocol significance
