2. EXISTING CHRONICLE COMPATIBILITY

Existing public Chronicle entries are locked and must remain canonical.

Do not rewrite them.

Do not delete them.

Do not silently replace them.

Do not duplicate them with derived entries.

If a derived institutional Chronicle Entry overlaps an existing locked entry:

* detect overlap,
* suppress duplicate rendering,
* or show it only as lineage/support metadata,
* never double-publish the same fact.

A) Existing locked entries remain primary; derived entries appear as “Institutional record support.”

B) Derived entries appear in a separate “Institutional Register feed” section below locked Chronicle.

4. CHRONICLE ENTRY STATUS DISCIPLINE

Only public-safe statuses may appear.

Rules:

* published locked manual entries = canonical public Chronicle.
* derived draft entries may be shown only as draft / institutional record support, not final Chronicle.
* held entries must not appear publicly except in labs.
* rejected entries must not appear publicly.
* superseded entries must be handled append-only if surfaced.

Do not present draft as final.

Do not imply governance approval.

Do not imply DAO approval.

──────────────────────────────

5. TIMELINE / CHRONOLOGY INTEGRATION

Use chronology only as factual ordering metadata.

Allowed:

* block number
* verified timestamp if available
* sequence position
* source transaction
* lineage link

Not allowed:

* story interpretation
* emotional framing
* AI-generated narrative
* speculative chapter meaning

Chronology orders.

Story interprets later.

Keep them separate.
