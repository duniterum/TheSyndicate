Core objective:

Build the minimal Chronicle Admission Layer:

Institutional Register Entry
→ Chronicle Admission Candidate

This sprint decides which verified institutional memories are eligible to become Chronicle entries later.

This sprint does NOT:

* auto-publish Chronicle entries
* rewrite existing Chronicle
* generate Story
* build Recognition
* build Member Register
* build Governance / DAO
* build AI
* change derivation rules
* create new contracts

It only creates the admission bridge between durable institutional memory and future Chronicle entries.

1. CHRONICLE ADMISSION OBJECT

Create ChronicleAdmissionCandidate.

Fields should include:

* id
* sourceInstitutionalRegisterEntryId
* sourcePromotionDecisionId
* sourceChronicleReviewCandidateId
* sourceMemoryCandidateId
* sourceSignalId
* sourceEventId
* sourceTxHash if available
* sourceBlock if available
* register = protocol-institutional
* category
* admissionStatus
* admissionReason
* proposedChronicleTitle
* proposedChronicleSummary
* proposedChronicleRegister
* verificationStatus
* lineage
* createdFrom

2. ADMISSION STATUS

Use strict statuses:

* admitted
* review
* held
* rejected

Rules:

* active verified institutional entries may become admitted or review.
* draft entries should usually remain review.
* held entries remain held.
* rejected entries are rejected or excluded.
* coverage-limited entries must not be admitted.

3. ADMISSION RULES

Define deterministic admission rules.

Suggested baseline:

Admit or review:

* verified protocol-birth facts
* token deployment
* sale deployment
* archive deployment
* initial distribution
* first liquidity
* first burn / Proof of Fire
* first verified protocol milestone
* major verified treasury action
* major verified operations action
* major verified liquidity action

Hold:

* coverage-limited facts
* facts requiring human context
* facts without complete lineage

Reject:

* routine operational noise
* repeated small movements
* facts with unsafe copy
* member-living entries
* unverified “first” claims

4. CHRONICLE COMPATIBILITY

Do not mutate existing Chronicle entries.

Do not add new public Chronicle entries yet.

Only prepare compatibility so a future sprint can convert:

ChronicleAdmissionCandidate
→ ChronicleEntry

Existing locked Chronicle entries remain untouched.

5. PROTOCOL-CENTRIC FRAMING

Institutional Chronicle must stay protocol-centric.

Allowed framing:

* protocol deployed
* protocol recorded
* protocol seeded
* protocol executed
* protocol reached
* protocol verified

Avoid:

* founder achievement
* founder made history
* legendary
* heroic
* investor reward
* profit
* return
* governance right
* financial promise

Founder/steward/system wallet actions may be admitted only when they are institutional protocol facts.

6. GENESIS_EVENT HANDLING

GENESIS_EVENT is allowed only as a verified institutional protocol-birth fact.

Unsafe genesis/historic marketing copy remains blocked.

Do not weaken the guardrails.
