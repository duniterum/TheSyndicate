LIVE-ERA PURCHASE UI CORRECTNESS V1

Goal

Make the Join / purchase UI derive purchase amounts, rate labels, minimums, and package outputs from the live Sale V2 era/rate truth instead of hardcoded Genesis assumptions.

This must happen before any corrected Sale V2 redeploy, because otherwise the UI will keep showing wrong package outputs when eras change.

Current confirmed truths

Contract truth:

* Sale V2 already has automatic era logic.
* currentEra changes automatically by seat count / cap condition.
* each era has its own minUsdc6.
* each era has its own synPerUsdc.
* quote() uses the live era and rate.
* buy() uses the same live era and rate.
* one-seat identity is already separate from repeat purchases.
* repeat purchases add SYN / recognition but do not create extra seats.
* deployed Genesis cap is the parameter mismatch, not the contract architecture.

Frontend truth:

* package display is still hardcoded to Genesis rate.
* LivePurchase uses `usdc * 100` for displayed SYN.

* package/rank cards are static.
* minSynOut fallback is unsafe for later eras because it falls back to Genesis rate.
