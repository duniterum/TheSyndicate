SPRINT 4 — CHRONICLE & RECOGNITION CONSTITUTION

Use `docs/canon/`, the Protocol Metrics Registry, the Protocol Event Pipeline, and the Protocol Memory foundation as source of truth.

This is an execution sprint.

No redesign.
No new contracts.
No referral contract.
No burn automation.
No governance.
No AI.
No marketplace.
No new tokenomics.

Goal:
Now that the protocol can see events and derive candidates, define the constitutional rules for what deserves memory and recognition.

Core distinction:
Activity = everything verified.
Chronicle = what mattered.
Recognition = who left a meaningful trace.

---

## PART 1 — CHRONICLE CONSTITUTION

Create or consolidate a doctrine file defining:

What qualifies for Chronicle?

Rules:

Always Chronicle-worthy:

* protocol launch / first member
* chapter sealed
* Proof of Fire #001
* first artifact mint
* first Patron Seal
* first LP event
* major protocol milestone
* founder action with protocol significance
* future member-number milestones: #100, #333, #1,000, #3,333, #10,000

Sometimes Chronicle-worthy:

* major purchase / rank milestone
* Patron Seal mint
* important burn after #001
* founder-funded Operations
* major Operations spend
* future referral milestone
* unusual protocol event
* external recognition / press / listing

Usually Activity-only:

* normal purchase
* normal transfer
* normal small NFT mint
* normal LP swap
* normal wallet balance update

Chronicle entry must include:

* title
* why it mattered
* source event(s)
* verification link
* legal-safe wording
* status: CANDIDATE / APPROVED / PUBLISHED / REJECTED
* whether automatic or operator-curated

Do not auto-publish everything.

---

## PART 2 — RECOGNITION CONSTITUTION

Define Recognition without creating a new rank system.

Recognition means:
A member did something visible, meaningful, supportive, or historically relevant.

Recognition must not mean:

* KYC
* verified real-world identity
* financial rights
* governance rights
* yield
* ROI
* guaranteed benefit

Supported display modes:

* anonymous: Member #217
* alias: user-chosen handle, later
* public name: self-declared only, non-KYC, later

Recognition candidates may come from:

* rank milestone
* Patron Seal
* Proof of Fire participation
* artifact mint
* important referral attribution later
* founder support action
* chapter participation
* report mention
* Chronicle mention

Do not implement paid alias/profile now unless already trivial.
Only define the rules and update existing candidate logic/docs if needed.
