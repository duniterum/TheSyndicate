Important architecture decision

The protocol journey backbone is:

Identity State
→ Next Best Action
→ Protocol Actions
→ CTA/UI

NOT:

Protocol Actions

→ later guidance

The selector must become the reusable source of journey logic.

A. Build a pure selector

Create:

src/lib/next-best-action.ts

Requirements:

* Pure TypeScript.
* No React.
* No hooks.
* No chain reads.
* No JSX.
* No rendering.
* No side effects.

Input:

A typed state object built from already-existing protocol state.

Output:

Ordered protocol action ids.

The selector should only consume already-computed state.

Never fetch data itself.

B. Use existing proven logic

Do not invent a new journey.

Start from the existing logic inside:

* CockpitNextMove
* CockpitProgression
* current member progression logic
* rank progression logic
* holder-index identity logic

The goal is extraction and normalization.

Not redesign.

Not reinvention.

C. Identity state model

Implement only states that are reliably detectable today.

Visitor

* wallet disconnected

Connected non-member

* connected
* no holder-index member record

Member

* holder-index member exists

Collector

* owns Archive1155 artifact(s)

Higher-rank member

* deterministic rank state

Do not implement:

* liquidity participant
* referrer
* claim eligible
* campaign participant
* merch participant

Those remain future states until reliable protocol data exists.

D. Output rules

The selector may only emit actions that are:

LIVE
or
LIVE-UNAUDITED

Never emit:

PENDING
PREVIEW
SEALED

Meaning:

No referral actions.
No campaign rewards.
No merch rewards.
No SeatRecord721.
No marketplace actions.

The selector must respect protocol-actions.ts.

Output must always be a subset of live registry actions.
